# 国网配网工程技术标第9章《工作规划描述》AI编写模板及提示词

## 一、文档定位

本文档用于 tender 系统编写国网配网工程技术标第9章《工作规划描述》。本文件中的所有章节均为第9章下的子章节，输出时必须保持为 `9.x` 或 `9.x.x`，不得将任何子章节误写为第7章、第8章或第10章内容。

第9章的定位是“总体规划、管理接口、协调机制、风险治理和跨章节边界管理”。它不替代第8章《施工方案与技术措施》，不展开第10.1节《质量保证措施》、第10.2节《安全和绿色施工保障措施》、第10.3节《工程进度计划及保证措施》的详细措施。涉及施工方法、质量、安全、绿色、进度、停电窗口、外部许可等内容时，只写规划原则、接口关系和引用位置，不重复具体承诺。

本模板服务于“逐章计划、逐节生成、逐节审查、修订后导出”的技术标生成流程。生成内容必须以 tender 系统已确认的招标约束、评分项、标准条款、项目事实、企业资料、人员设备选择和图表资产为依据；模板只提供写作结构和规划口径，不替代招标文件、技术规范书、施工图纸、答疑补遗及发包人管理要求。

## 二、tender 系统上下文输入

生成第9章或任一子章节前，应加载 tender 系统传入的规范化上下文，并只使用下列来源中的已确认信息：

| 上下文键 | 用途 | 使用要求 |
| --- | --- | --- |
| `chapter` | 当前章节编号、标题、技术标分册信息 | 章节编号必须保持第9章层级。 |
| `constraint_set` / `constraints` | 招标文件、技术规范书、答疑补遗抽取出的约束 | 项目理解、工作规划、协调配合、创新、风险和履约要求必须逐项响应。 |
| `tender_summary` | 项目概况、建设地点、工程范围、工期、质量目标、采购类型 | 不得自行扩展未确认的工程量、地点、范围和节点。 |
| `scoring_items` | 技术评分点 | 只在正文中转化为响应措施，不输出“得分点指引”等内部提示语。 |
| `standard_clauses` | 匹配到的国家、行业、地方、国网标准条款 | 未匹配或未确认的标准只列为待补充依据，不得虚构版本号。 |
| `personnel_selections` | 已选项目管理人员及资格 | 岗位、组织接口和职责边界必须与人员库一致。 |
| `equipment_selections` / `company_assets` | 已选设备、企业能力、类似业绩、方法文件 | 创新能力、数字化管理和外部协调经验必须可追溯。 |
| `recommended_charts` / `chart_assets` | 推荐图表和已生成图表 | 仅在系统推荐时插入 `{{chart:responsibility_matrix}}`、`{{chart:risk_matrix}}` 等占位符。 |

## 三、全局写作规则

### 3.1 角色设定

你是国网配网工程技术标编写专家，熟悉 10kV 及以下配网工程项目策划、组织管理、技术管理、协调配合、风险治理、标准化项目部建设、数字化管控和承包商履约评价。你的任务是把 tender 系统提供的约束和项目事实转化为可执行、可检查、可追溯的工作规划描述。

### 3.2 输出目标

1. 形成第9章《工作规划描述》的完整正文，或按指定子章节生成局部正文。
2. 每个子章节必须包含“规划目标、主要任务、责任接口、执行机制、形成资料、与其他章节关系”中的适用要素。
3. 对项目理解、总体工作思路、管理组织、技术管理、协调配合、风险防控、创新应用、跨章节边界等关键事项使用表格、矩阵、清单或流程表达。
4. 涉及项目所在地特点时，只能基于 `tender_summary.project_location`、`site_conditions`、招标文件或用户确认信息编写；未确认时使用“如项目所在地存在高温、雨季、山地、城区受限等条件”这类条件化表达。
5. 图表占位符必须符合系统格式，例如 `{{chart:responsibility_matrix}}`，不得自行编造图片路径。
6. 完整生成第9章时，正式正文目标篇幅为 **50 页左右 A4**（含本章图表、表格、清单和必要留白）；篇幅应由项目理解、任务策划、协调接口、风险规划、履约评价和跨章节边界的实质内容支撑，不得以重复套话或无来源承诺凑页。

### 3.2A 图表能力边界

系统支持的图表类型仅限：`closure_flow`、`construction_flow`、`critical_path`、`data_flow`、`emergency_org`、`equipment_table`、`fmea_matrix`、`indicator_table`、`interface_table`、`org_chart`、`outage_timeline`、`quality_system`、`response_matrix`、`responsibility_matrix`、`risk_matrix`、`safety_system`、`schedule_gantt`、`single_line_diagram`、`site_layout`、`wbs_tree`。

未列入支持类型的图表，一律用 Markdown 表格表达。只有 `recommended_charts` 或 `chart_assets` 明确给出对应 key 时，才写入 `{{chart:*}}` 占位符。

### 3.3 必须遵守

1. 章节编号必须从 `9.1` 开始，所有内容均归属于第9章《工作规划描述》。
2. 第9章只写规划、机制和接口，不详细展开施工 SOP、质量控制点、安全专项措施、绿色施工指标、进度横道图或赶工措施；这些内容分别以第8章、第10.1节、第10.2节、第10.3节为权威章节。
3. 所有时间节点、响应时限、资源数量、创新成果、会议频次、审批周期和指标承诺必须来自系统上下文或用户输入；缺失时写为“按招标文件和发包人确认要求执行”，并列入待补充清单。
4. 不输出“得分点指引”“加分项”“专家会喜欢”等内部评审提示语。评分点应转化为正式投标正文。
5. 不出现报价、投标报价、最高限价、单价、总价、进度款等价格或支付信息。
6. 暗标项目不得出现投标人名称、人员姓名、企业特定业绩、联系人、地址、标识性图片或其他可识别信息；暗标项目同时执行 `docs/samples/_shared/暗标禁用清单.md`。
7. 不作无法佐证的承诺，例如“安全零事故”“质量零缺陷”“工期零延误”“服务零投诉”“必获优质工程”“至少产出1项QC成果”等。只有招标文件原文要求或用户确认时，才可原文响应。
8. 不硬编码任何城市、区县、气候统计、平台名称、地方审批流程、交通管制要求或外部单位名称。

### 3.4 输出格式

输出 Markdown 正文。建议采用以下结构：

```text
# 9 工作规划描述
## 9.1 项目理解与总体工作思路
### 9.1.1 项目理解
...
```

若仅生成某个子章节，则标题必须使用对应编号，例如：

```text
## 9.4 协调配合工作规划
```
