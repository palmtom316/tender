### 2.1 第8章《施工方案与技术措施》提示词 —— 评级：B+（基本可用，增补可达 A-）

### 2.2 第9章《工作规划描述》提示词 —— 评级：A-（推荐采用）

**优点**

1. 文档定位明确指出"第9章只写规划、机制和接口"，并把第8章施工方法、第10.1质量控制、第10.2安全绿色、第10.3进度计划列为"权威章节"——这是上一轮提示词体系里**最规范的章节边界约定**。
2. 9.8"跨章节协同与边界管理"用一张"跨章节索引表"统一约定主题与权威章节的对应关系，避免投标文件跨节冲突——这一节本身就是国网评标常见"投标文件自相矛盾"的减分点的有效防御。
3. 9.3"项目管理组织与制度规划"明确指明"会议频次、报告时限、审批周期必须有来源；缺失时写'按发包人确认管理要求执行'"——精准命中评委审查制度类条款时常用的盘问点。
4. 9.4"协调配合工作规划"禁用"承诺未经确认的停电窗口、夜间施工许可、占道许可、交通管制时段、绿色通道审批、外部机构响应时限"——与 10.2.6、10.3.9 的"外部许可禁止承诺"形成跨章节一致约束。
5. 9.6 风险防控不允许 LLM 自行打分（"风险等级、阈值和处置时限必须有来源；缺失时只写风险等级说明，不自行打分"），杜绝了 LEC 数值无源化。

### 3.2 渲染引擎缺陷（按严重度排序）

#### CR-1（严重）：flow 类回退渲染只画"垂直链"，忽略 edges 拓扑

**证据**：`renderers.py` 第99-117 行 `_render_flow_svg`：

```python
for index, node in enumerate(spec.nodes):
    y = top + index * row_height
    parts.append(_rect(left, y, box_width, box_height, "#eef6ff", "#6d93b8", 6))
    parts.append(_text(node.label, left + box_width / 2, y + 25, 15, anchor="middle", weight="600"))
    if index < len(spec.nodes) - 1:
        x = left + box_width / 2
        parts.append(_line(x, y + box_height, x, y + row_height - 8, "#6d93b8"))
        parts.append(_arrow(x, y + row_height - 8, "#6d93b8"))
```

`spec.edges`（FlowChartSpec 字段）在 fallback 渲染时**完全未被使用**——只是把所有节点纵向排列，并在相邻节点之间画一根线。

**评标后果**：
- `quality_system / safety_system / emergency_org / org_chart` 的拓扑通常是"1 个项目经理 → 3~5 个职能负责人 → 5~10 个班组"的树状结构，fallback 渲染出来是"项目经理-技术负责人-质量负责人-安全负责人-...-班组长"一根直线，**无法体现层级与并行**。
- 评标专家看到"质量管理体系图"是直线，会直接质疑投标人组织能力或制图能力。
- Mermaid sidecar 不可用时（部署环境无 sidecar / 网络异常 / 超时），所有体系图都会进入此 fallback。

**修复建议**：fallback 渲染器实现真正的树状/DAG 布局（如简化版 Reingold-Tilford 算法），或当 sidecar 不可用时**拒绝渲染**并标记 `needs_review`，避免输出错误的图。

#### CR-2（严重）：甘特图回退渲染不画依赖、不画时间刻度

**证据**：`renderers.py` 第120-143 行 `_render_gantt_svg`：

```python
parts.append(_line(chart_left, top - 20, chart_left + chart_width, top - 20, "#9aa4af"))
for index, task in enumerate(spec.tasks):
    ...
    parts.append(_rect(x, y + 5, bar_w, 18, "#dff0d8", "#78a66a", 4))
    parts.append(_text(f"{task.start.isoformat()} - {task.end.isoformat()}", x + bar_w + 8, y + 19, ...))
```

- `spec.dependencies`（GanttChartSpec 字段，最多 120 条）**完全未渲染**，"任务间依赖箭头"缺失。
- 横轴**只有一根分隔线**（line 132），没有"年/月/周/日"刻度，评标专家看不出工期跨度与里程碑节点位置。
- 没有 `section` 分组渲染。

**评标后果**：
- 第10.3.5 节是评标专家最关心的图，**缺少依赖箭头**导致"关键路径"不可视化；
- **缺少时间刻度**导致评标专家无法直观判断"开工→送电"工期是否合理、里程碑是否对齐招标要求。

**修复建议**：
- 增加横轴时间刻度（按工期跨度自适应"日/周/月"）；
- 渲染 dependencies 为虚线箭头；
- 渲染 section 分组背景色或左侧分组标签；
- 高亮关键路径任务（spec 中标记 `is_critical: true` 的任务用红色边框）。

#### CR-3（中）：风险矩阵单元格文本截断

**证据**：`renderers.py` 第168-171 行 `_render_risk_matrix_svg`：
```python
if cell and cell.items:
    lines = _wrap("；".join(cell.items), 12, 4)
    for line_index, line in enumerate(lines):
        parts.append(_text(line, x + 10, y + 24 + line_index * 18, 12, anchor="start"))
```
- `_wrap(text, 12, 4)`：每行 12 字、最多 4 行，超出截断为省略号；
- 单元格最多列出 8 个 items（`RiskCell.items` max_length=8）；
- 当 8 个风险项叠加时，4 行 × 12 字 = 48 字无法容纳，大量风险项被省略号吃掉。

**评标后果**：评委关注的"次重大风险"可能正好被截断，导致评分点漏写。

**修复建议**：当 items 数量超过 3 个时，单元格只显示"风险数量 + 最高风险简称"，详情通过表格在正文展开。

#### CR-4（中）：图号生成规则不符合国网技术标惯例

**证据**：`captions.py` 第33-37 行：
```python
def _chapter_segment(chapter_code: str | None) -> str:
    if not chapter_code:
        return "1"
    match = re.search(r"\d+", chapter_code)
    return match.group(0) if match else "1"
```
- 取章号首段数字，`"10.1.3"→"10"`，输出"图10-1"；
- 国网技术标评审惯例为"图10.1-1"（章号保留二级，序号在末尾），便于按目录索引。

**评标后果**：评标专家手动按目录索引图表时，"图10-1"找不到归属节，需要逐张图回查图名。

**修复建议**：`_chapter_segment` 改为返回完整章号（如 `10.1`、`10.3`），格式 `图10.1-1`；遇到 `10.1.3` 时按 `10.1` 编号，遇到 `10.3.5` 时按 `10.3` 编号。

#### CR-5（中）：AI 生成 spec 未做暗标脱敏

**证据**：`chart_generation_service.py` 第241-251 行：
```python
user_content = {
    "chart_type": chart_type,
    "title": title,
    "placeholder_key": placeholder_key or f"{chart_type}_main",
    "context": context,  # 整个 context 上行
}
```
context 中可能包含 `personnel_selections.snapshot_json`（含姓名、证书号）、`tender_summary.project_location`（地点）、`company_profiles.company_name`（企业名称）等暗标敏感字段。

**评标后果**：暗标项目下，AI 网关返回的 spec 中可能复用上行的姓名、地点、企业名称作为节点标签，导致暗标违规。

**修复建议**：
- 在上行前由 `_redact_context_for_chart` 脱敏：暗标项目下移除/泛化 personnel.name、company_name、project_location、tender_no 等字段；
- 在 spec 校验时增加"暗标关键词扫描"，命中投标人名称/地点等关键词时拒绝写入。

#### CR-6（低-中）：default_chart_spec 回退过于简陋

**证据**：`chart_generation_service.py` 第188-224 行：
```python

        **base,
        "tasks": [
            {"id": "prepare", "label": "施工准备", "start": "2026-06-01", "end": "2026-06-05", "group": "准备阶段"},
```

- 甘特图回退仅 2 个任务、施工流程回退仅 3 个节点；
- 默认日期 `2026-06-01` 硬编码，在 5 月生成时显示为"未来项目"是合理的，但下半年生成时会出现"过期日期"。

**评标后果**：AI 生成失败时落入此回退，输出明显非投标级图。

**修复建议**：
- 回退 spec 状态默认为 `needs_review` 而非 `draft`，强制人工复核；

5. **建立"图表素材准入审批"**：图表 spec 必须 `approved` 才能在 formal 投标包中插入（现有 `chart_asset_injector._formal=True` 已实现，但需在前端工作流明确)。

## 八、特别说明（针对图表能力）

本报告对图表代码的评价较为严厉，原因在于：

1. **配网评标对图表的依赖度高**：评标专家阅卷时，"组织架构、关键路径、风险矩阵、应急流程"是快速判断投标人专业水平的核心信号。图表错位或缺失，会直接影响主观分。
2. **当前代码层的"承诺-能力"剪刀差**：提示词向 LLM 承诺了 70+ 处图表/表格图位，代码层 e2e 实际可插入 ≈ 14 处占位且仅 4 类图表能稳定渲染（quality_system / safety_system / risk_matrix / responsibility_matrix 中的 3 个矩阵类基本可用，flow 类回退渲染存在缺陷），这种剪刀差不解决，五份提示词的工程化优势将被图表层的瓶颈抹平。
3. **Mermaid sidecar 是单点依赖**：如果 sidecar 不可用，所有 flow / gantt 图都进入有缺陷的 fallback。生产环境必须保障 sidecar SLA，或重写 fallback。
4. **暗标脱敏是评标硬伤**：暗标项目下 AI 网关传递的 spec 中若出现可识别信息，可能直接构成"暗标违规"——这是评委可现场判定的废标项。

**优先建议**：在下一迭代中以"图表能力补强"为主题，集中处理 6.2.P0 + 6.2.P1.第5项的图表类型扩展。完成后再做一次端到端抽样：用 5 份提示词 + 补强后的图表能力跑 3~5 个真实招标文件，由评标专家对生成稿盲评图表完整度与可读性，进一步收敛。
