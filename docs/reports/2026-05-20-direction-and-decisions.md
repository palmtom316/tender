## 2. 模板爆炸问题

**问题：** tender 当前按"工程类型 × 业主"维度组织模板，行业一多就维护不下去。

**关键洞察（用户提出）：**
- 技术标核心章节高度收敛：技术应答 / 质量 / 工期 / 安全文明 / 绿色 / 售后 / 现场管理 ≈ **7 个稳定章节**。
- 商务标核心章节也收敛：公司资质 / 公司业绩 / 人员资格业绩 / 公司实力 ≈ **4 个稳定章节**。
- **11 个核心章节策略 + 招标文件抽取 + 业主偏好** 即可覆盖 80% 工程标书。

**共识：**
1. **不再"按工程类型分模板"，改"按章节角色分策略"**。
2. 模板这个粗对象拆成三层：**目录骨架**（数据/配置）+ **章节策略**（写作菜谱）+ **行业语料**（片段库）。
3. 模板数量从 N × N 收敛到 ~30 个稳定策略。

**4 个必须补的偏差：**

| 偏差 | 修正 |
|---|---|
| 业主偏好走 LLM 知识库不可靠 | 走本地 ledger 沉淀；LLM 起草，本地数据兜底 |
| 7+4 之外的格式化小章节（投标函/承诺书）缺失 | 加"必填固定附件集合"，由招标文件抽取触发 |
| "技术要求应答"方差最大 | 裂成 4-5 个子策略：施工方案型 / 参数应答表型 / 实施方案型 / 设计方案型 |
| 评分权重决定章节详略 | 每章 3 档（light/standard/deep），按 `scoring_criteria` 抽出来的权重自动选 |

## 3. 规范库还要不要做

**问题：** 大模型已经知道很多国标，tender 当前的 AI 写作也没用到入库的规范——还要继续做规范入库吗？

**共识：保留，但重新定位。**
- **错误定位：** 规范库 = AI 的知识来源（→ 应该删，LLM 已经知道）。
- **正确定位：** 规范库 = "防 AI 伪造条文号"的反查源，跟业绩、人员、固定承诺同等级。

**理由：**
1. 标书引用规范必须字字精确（GB 50300-2013 第 5.0.2 条，错一位就是大错）。
2. LLM 会编造条文号和原文——这在标书里是致命的。
3. 地方/行业/企业标准（DBJ/T、DL/T、Q/GDW、Q/CSG）LLM 基本不知道。

**"AI 写作没用到规范"是工程问题，不是定位错误。三件事要修：**
- 规范库实际质量（可能空或解析烂）。
- `_match_standard_clauses` 阈值过严（`_score >= 0.35` 是 token overlap，召回率低）。
- prompt 没强制"必须引用本地 standard_clause_id"。

## 4. 目录骨架怎么生成

**用户提议：** 解析招标文件 → 找强制响应内容 → 由 AI 根据业主要求补全 → 形成目录骨架。

**两个致命漏洞：**
1. **招标文件不会写"行业默契"章节**（如施工组织设计、安全文明措施等，招标文件可能只说"按规范"，但少了就废标）。
2. **AI 自由补的部分无法验证完整性**——没有 ground truth → 不能跑规则 → 不能 gate。

**修正方案：双源合流 + LLM 只做编排**

```
A. 招标文件抽取（硬约束）
   - 投标文件格式必填项
   - 正文"应/须/必须"响应点
   - 评分点权重
B. 行业章节图谱（数据，不是模板）
   - 节点 + 属性 + tag
   - 全行业 100-200 节点
C. 业主偏好库（可选，长期沉淀）
   ↓
LLM 编排（不创作，只重组）
   - 删除不适用 / 注入子要求 / 命名贴合 / 按权重选档
   ↓
规则化校验 → 用户确认 → 最终目录
```

**核心原则：LLM 做"选择题和排序题"很可靠，做"填空题"会编。** 把目录生成约束成选择题（从图谱里挑），LLM 是可控的。

## 5. 业主偏好 ledger 冷启动问题

**问题：** 业主偏好 ledger 没积累怎么办？

**共识：业主偏好的真实权重被高估，约只占标书质量决定因素的 5-10%。冷启动不该卡在它身上。**

**三层兜底（从可信到不可信）：**

| 层 | 数据源 | 实现成本 |
|---|---|---|
| L1：本招标文件本身的格式特征 | 抽 fingerprint 即可 | 极低 |
| L2：行业默认值（11 章策略 + 五段论） | 已有 | 0 |
| L3：LLM 推测 + 强制标记 `source: "llm_speculation"` | 用户必须确认才入库 | 中 |

**ledger 自然积累路径：**
```
首次跑某业主招标 → 存 fingerprint
第二次跑同业主 → 与历史 fingerprint 比对，提取高频共现项 → 候选 ledger 条目
人工确认 → 进入 confirmed ledger
第三次起 → 自动注入生成 prompt
```

**产品哲学：第一次给某业主写 = AI 必然平庸；第二次开始能学到东西。这是诚实的产品定位，不是 bug。**

## 6. 系统如何自行迭代

**澄清：自行迭代 ≠ "AI 自己改自己"。** 真正能跑通的定义：

> 系统自动累积学习信号 + 自动产出建议 + 人工确认后更新数据资产

**三层循环：**

| 层 | 周期 | 数据量 | 动作 | 关键点 |
|---|---|---|---|---|
| L1：项目内 | 秒-小时 | 单项目 | 沉默收集（fingerprint / 编辑 diff / 触发的 finding / 页数偏差） | 不做任何"学"的动作，只把信号结构化存下 |
| L2：业主级 | 月-季度 | 2-N 个项目 | 比对历史 fingerprint，抽稳定项作为 ledger 候选 | 数据规则驱动（≥70% 共现），不让 LLM 判 |
| L3：系统级 | 季度-年 | 几百项目 | 跨业主跨行业模式挖掘 → 季度调优建议报告 | AI 帮人发现该学什么，运营评审决定 |

**优先级（必须按此顺序）：**
1. L1 信号收集（**这是基石**，光是回看"AI 哪里写得烂"就够本）。
2. L2 业主 ledger（等单业主跑过 3+ 项目再启动）。
3. L3 调优报告（数据积累 6 个月+ 再启动）。

**反过来做必败。**

**tender 现有可复用基础设施：**
- `tender_constraint_set.version` 已有版本化心智。
- `bid_generation_subsection_run`（prompt_hash, model, tokens）—— L1 关键数据已采。
- `compliance_check_finding`（rule_code, status, waived_by）—— L3 误报率信号现成。
- `chapter_draft.generation_rounds`—— 重写次数已记。

**关键缺口：**
- **用户编辑 diff 的结构化存储**（L1 基石，最该补）。
- **`tender_document_fingerprint` 表**（L2 入口）。
- **`owner_preference_ledger`**（candidates + confirmed）。

## 7. 解析能力审计结论

详见 `docs/reports/2026-05-20-tender-parsing-readiness-audit.md`。一句话：

**tender 解析的"硬件基础"非常扎实（MinerU 2.7 + 完整 source 反查链 + 多档 AI planner），但"软件能力"被深度绑定到了国网配网施工标这一个垂直场景上。**

→ 直接含义：
- **Ledger / Evidence / Gap Engine 主线（plan 中那 5 步）—— 解析侧基本够用，不需要先动解析**。
- **通用 11 章方案会立即被三个抽取短板卡住**：评分子项抽取不够细 / 投标文件目录没专项解析器 / 业主偏好 fingerprint 完全没采。
