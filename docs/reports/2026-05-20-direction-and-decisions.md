## 2. 模板爆炸问题

**问题：** tender 当前按"工程类型 × 业主"维度组织模板，行业一多就维护不下去。

**关键洞察（用户提出）：**
- 技术标核心章节高度收敛：技术应答 / 质量 / 工期 / 安全文明 / 绿色 / 售后 / 现场管理 ≈ **7 个稳定章节**。
- 商务标核心章节也收敛：公司资质 / 公司业绩 / 人员资格业绩 / 公司实力 ≈ **4 个稳定章节**。
- **11 个核心章节策略 + 招标文件抽取 + 业主偏好** 即可覆盖 80% 工程标书。

**共识：**
1. **不再"按工程类型分模板"，改"按章节角色分策略"**。
2. 模板这个粗对象拆成三层：**目录骨架**（数据/配置）+ **章节策略**（写作菜谱）+ **行业语料**（片段库）。
3. 模板数量从 N × N 收敛到 ~30 个稳定策略。

**4 个必须补的偏差：**

| 偏差 | 修正 |
|---|---|
| 业主偏好走 LLM 知识库不可靠 | 走本地 ledger 沉淀；LLM 起草，本地数据兜底 |
| 7+4 之外的格式化小章节（投标函/承诺书）缺失 | 加"必填固定附件集合"，由招标文件抽取触发 |
| "技术要求应答"方差最大 | 裂成 4-5 个子策略：施工方案型 / 参数应答表型 / 实施方案型 / 设计方案型 |
| 评分权重决定章节详略 | 每章 3 档（light/standard/deep），按 `scoring_criteria` 抽出来的权重自动选 |

## 3. 规范库还要不要做

**问题：** 大模型已经知道很多国标，tender 当前的 AI 写作也没用到入库的规范——还要继续做规范入库吗？

**共识：保留，但重新定位。**
- **错误定位：** 规范库 = AI 的知识来源（→ 应该删，LLM 已经知道）。
- **正确定位：** 规范库 = "防 AI 伪造条文号"的反查源，跟业绩、人员、固定承诺同等级。

**理由：**
1. 标书引用规范必须字字精确（GB 50300-2013 第 5.0.2 条，错一位就是大错）。
2. LLM 会编造条文号和原文——这在标书里是致命的。
3. 地方/行业/企业标准（DBJ/T、DL/T、Q/GDW、Q/CSG）LLM 基本不知道。

**"AI 写作没用到规范"是工程问题，不是定位错误。三件事要修：**
- 规范库实际质量（可能空或解析烂）。
- `_match_standard_clauses` 阈值过严（`_score >= 0.35` 是 token overlap，召回率低）。
- prompt 没强制"必须引用本地 standard_clause_id"。

## 4. 目录骨架怎么生成

**用户提议：** 解析招标文件 → 找强制响应内容 → 由 AI 根据业主要求补全 → 形成目录骨架。

**两个致命漏洞：**
1. **招标文件不会写"行业默契"章节**（如施工组织设计、安全文明措施等，招标文件可能只说"按规范"，但少了就废标）。
2. **AI 自由补的部分无法验证完整性**——没有 ground truth → 不能跑规则 → 不能 gate。

**修正方案：双源合流 + LLM 只做编排**

```
A. 招标文件抽取（硬约束）
   - 投标文件格式必填项
   - 正文"应/须/必须"响应点
   - 评分点权重
B. 行业章节图谱（数据，不是模板）
   - 节点 + 属性 + tag
   - 全行业 100-200 节点
C. 业主偏好库（可选，长期沉淀）
   ↓
LLM 编排（不创作，只重组）
   - 删除不适用 / 注入子要求 / 命名贴合 / 按权重选档
   ↓
规则化校验 → 用户确认 → 最终目录
```

**核心原则：LLM 做"选择题和排序题"很可靠，做"填空题"会编。** 把目录生成约束成选择题（从图谱里挑），LLM 是可控的。

## 5. 业主偏好 ledger 冷启动问题

**问题：** 业主偏好 ledger 没积累怎么办？

**共识：业主偏好的真实权重被高估，约只占标书质量决定因素的 5-10%。冷启动不该卡在它身上。**

**三层兜底（从可信到不可信）：**

| 层 | 数据源 | 实现成本 |
|---|---|---|
| L1：本招标文件本身的格式特征 | 抽 fingerprint 即可 | 极低 |
| L2：行业默认值（11 章策略 + 五段论） | 已有 | 0 |
| L3：LLM 推测 + 强制标记 `source: "llm_speculation"` | 用户必须确认才入库 | 中 |

**ledger 自然积累路径：**
```
首次跑某业主招标 → 存 fingerprint
第二次跑同业主 → 与历史 fingerprint 比对，提取高频共现项 → 候选 ledger 条目
人工确认 → 进入 confirmed ledger
第三次起 → 自动注入生成 prompt
```

**产品哲学：第一次给某业主写 = AI 必然平庸；第二次开始能学到东西。这是诚实的产品定位，不是 bug。**

## 6. 系统如何自行迭代

**澄清：自行迭代 ≠ "AI 自己改自己"。** 真正能跑通的定义：

> 系统自动累积学习信号 + 自动产出建议 + 人工确认后更新数据资产

**三层循环：**

| 层 | 周期 | 数据量 | 动作 | 关键点 |
|---|---|---|---|---|
| L1：项目内 | 秒-小时 | 单项目 | 沉默收集（fingerprint / 编辑 diff / 触发的 finding / 页数偏差） | 不做任何"学"的动作，只把信号结构化存下 |
| L2：业主级 | 月-季度 | 2-N 个项目 | 比对历史 fingerprint，抽稳定项作为 ledger 候选 | 数据规则驱动（≥70% 共现），不让 LLM 判 |
| L3：系统级 | 季度-年 | 几百项目 | 跨业主跨行业模式挖掘 → 季度调优建议报告 | AI 帮人发现该学什么，运营评审决定 |

**优先级（必须按此顺序）：**
1. L1 信号收集（**这是基石**，光是回看"AI 哪里写得烂"就够本）。
2. L2 业主 ledger（等单业主跑过 3+ 项目再启动）。
3. L3 调优报告（数据积累 6 个月+ 再启动）。

**反过来做必败。**

**tender 现有可复用基础设施：**
- `tender_constraint_set.version` 已有版本化心智。
- `bid_generation_subsection_run`（prompt_hash, model, tokens）—— L1 关键数据已采。
- `compliance_check_finding`（rule_code, status, waived_by）—— L3 误报率信号现成。
- `chapter_draft.generation_rounds`—— 重写次数已记。

**关键缺口：**
- **用户编辑 diff 的结构化存储**（L1 基石，最该补）。
- **`tender_document_fingerprint` 表**（L2 入口）。
- **`owner_preference_ledger`**（candidates + confirmed）。

## 7. 解析能力审计结论

详见 `docs/reports/2026-05-20-tender-parsing-readiness-audit.md`。一句话：

**tender 解析的"硬件基础"非常扎实（MinerU 2.7 + 完整 source 反查链 + 多档 AI planner），但"软件能力"被深度绑定到了国网配网施工标这一个垂直场景上。**

→ 直接含义：
- **Ledger / Evidence / Gap Engine 主线（plan 中那 5 步）—— 解析侧基本够用，不需要先动解析**。
- **通用 11 章方案会立即被三个抽取短板卡住**：评分子项抽取不够细 / 投标文件目录没专项解析器 / 业主偏好 fingerprint 完全没采。

## 9. 通用解析系统的五项要求与可行性

**用户提出：** 想先把解析做成通用招标文件解析系统，要求：
1. 通用招标文件解析（不限行业）
2. 操作员只用 是/否 回答
3. 问题精排，卡片化显示
4. 商业 AI 模型普适性（不依赖单一模型能力）
5. 冗余信息合并

**评估：每条单点都可做，五条放在一起触发互斥。**

| 要求 | 难度 | 可行性 | 真正卡点 |
|---|---|---|---|
| 通用解析 | ⭐⭐⭐⭐ | 80% 主流可达，100% 任何招标永不可达 | 现有关键词被国网行业污染 |
| 是/否极简 | ⭐⭐⭐⭐ | 90% 决策可压，剩余 10% 压不了 | 冲突/数值/特殊说明本质不是是/否 |
| 卡片精排 | ⭐⭐ | 技术无难度 | 难在排序的 ground truth |
| 模型普适 | ⭐⭐⭐⭐ | 主流模型间互换可做，任何模型不可做 | 物理学问题：弱模型 hallucinate 概率高 |
| 冗余合并 | ⭐⭐⭐ | 规则级已有，语义级要补 | 跨文件 embedding 合并未做 |

**三对最关键的内在冲突：**
- 通用化 ↔ 精度（越通用越难精）
- 是/否极简 ↔ 信息密度（越简化越丢信息）
- 模型普适 ↔ 抽取质量（越普适越靠规则越精度低）

## 11. 选 B 之后的关键含义

### ① tender 当前架构其实就是 B 路线（重要事实）

回头看现有代码，专家工作流的骨架已经在了：

| 现有能力 | B 路线含义 |
|---|---|
| `tender_constraint_service` 有 reviewing/accepted/confirmed 状态流转 | 专家审核工作流已有 |
| `requirement_repo.confirm` / `reject` / `merge` | 专家有否决权、合并权 |
| `compliance_check_finding.waived_by` | 专家可豁免规则告警 |
| `bid_generation_subsection_run` 完整审计（prompt_hash / model / token / latency） | 专业级追溯已有 |
| `ad_hoc_chapter_task_card` 工作流（draft_ready 等状态） | 专家任务卡心智已有 |
| `requirement_matching` 输出 `requires_human_confirm` 标记 | 默认就是"AI 给建议，人做决定" |

**B 路线下，tender 不需要改方向，需要的是"沿着这条线把缺的部分补齐"**。

### ② 通用解析系统从"底座项目"降级为"持续工作"

B 用户能容忍人工修订（这是他们的本职工作），所以"通用解析必须先做完"不再成立。

### ③ 解析侧目标翻转：精度 > 召回

专家最烦的不是"AI 没抽出来"，是"AI 抽了一堆假阳性让我清"。所以：
- 弱信号宁可不抽（让专家自己看到）
- 抽出来的每一条都必须可解释（page / chunk_id / prompt_version / confidence）
- 多模型投票分歧的条目直接进人工，不强行融合

### ④ 不再追求是/否极简 → 工作流变成多人协作

大公司投标团队是商务/技术/造价分工的，真正的产品功能要求是：
- **角色权限**：商务专员只看商务标、技术专员只看技术标
- **任务分派**：评审人 A 审资格、B 审评分
- **评审签字**：合规留痕（`compliance_check_finding.waived_by` 已有字段，但还没多角色权限）
- **章节级编辑锁与版本对比**：避免多人覆盖

### ⑤ 行业专业字段必须保留并加强

B 用户不怕看到行业术语，反而要：
- 资质等级标准枚举（一/二/三级、承装承修承试 X 级、电力工程总包等级）
- 业绩字段行业化（kV / 容量 MVA / 回路数 / 距离 km / 合同金额）
- 人员证书细分（特种作业操作证、进网作业许可证、安规复训）
- 评分办法精细化（综合评分法 / 合理低价法 / 最低评标价法）

`docs/reports/2026-05-18-distribution-tender-full-coverage-gap.md` §四 已列出——B 路线下就是要把这些做扎实。

### ⑥ 模型普适性不再是强需求

需求降级为"主流商业模型顶配级（Claude Opus / GPT-5 / DeepSeek v4 Pro / Qwen3-Max）可切换"。`ai_extraction_planner` 现有分档策略基本够用。

## 12. B 路线下的明确路线图

```
Phase 1（沿现有架构加固，3-4 个月）
├── Requirement-Evidence Ledger plan 落地（已写好 5 步）
├── 多人协作工作流：角色权限 + 任务分派 + 章节级编辑锁

└── L1 信号收集：tender_document_fingerprint + chapter_draft_revision

Phase 2（解析能力升级，4-6 个月）
├── 投标文件目录专项解析器（精度优先，专家可调阈值）
├── 评分子项级抽取（精确到 5 分/3 分这种子项）
├── 跨文件 embedding 语义合并（让专家少看重复）

├── 业主 fingerprint 自动积累（fingerprint 比对 → 候选偏好 → 人工确认）
├── 章节策略命中率统计 + 季度调优建议报告
└── 行业知识图谱（11 章策略 + 各行业 tag 扩展）
```

每个 Phase 都是"现有架构的延展"，不是推倒重来。

## 13. 选 B 之后必须面对的商业反问

B 路线的目标客户（大公司投标团队、专业招投标专员）**已经有成熟工具**：标证通、千里马、北京筑龙、彩虹建标等。这些客户对工具切换极度保守（迁移成本极大、决策周期长、合规风险高）。

**tender 的差异化是什么？候选答案：**

| 差异化方向 | 含金量 | 难点 |
|---|---|---|
| AI 加持的版本（解析快、写作快） | ⭐⭐ | 老牌厂商也在加 AI，差异化窗口在收窄 |
| 全流程闭环（解析→评审→写作→审计→导出） | ⭐⭐⭐ | tender 现有架构有这个潜质 |
| 真正的合规审计链（每条都可反查） | ⭐⭐⭐⭐ | tender 的 source_chunk_id + audit log 是核心资产 |
| 行业纵深（电力配网做到最深，然后横向扩） | ⭐⭐⭐⭐ | 已经在做配网，是最有积累的方向 |
| 跨项目沉淀 + 自行迭代（同业主越用越准） | ⭐⭐⭐⭐⭐ | 老牌厂商做不到（他们没 fingerprint 心智） |

**初步倾向：差异化 = "全流程闭环 + 反查级合规 + 跨项目自迭代"，行业纵深从配网先深做。** 这三件事老牌厂商短期补不齐，且都是 tender 现有架构的延伸。

> **但这是商业判断，不是技术判断。最终决定权在用户。这是明天首要讨论的问题。**
