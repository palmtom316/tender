review 报告在业务线建模、符合性检查独立成层、工作流生命周期补全、扩展接口预留四个方向上的诊断是准确的，其修订建议清单（§6）可以直接作为修订基线。

## 1. 招投标法律法规合规要件（必补）

tender 系统产出的是正式投标文件，必须满足以下法律法规的合规底线。当前三份文档均未显式引用任何法规。

### 1.1 法律引用清单

| 法规 | 与 tender 直接相关的条款 |
|---|---|
| 《中华人民共和国招标投标法》(2017修正) | 投标文件编制（§27）、递交（§28）、开标（§34-36）、评标中标（§37-45） |
| 《招标投标法实施条例》(2019修正) | 投标保证金（§26）、投标有效期（§25）、废标情形（§51）、串标认定（§39-42） |
| 《电子招标投标办法》(2013) | 电子投标文件编制（§26）、电子签名（§27）、加密递交（§28）、开标解密（§29-31） |
| 《必须招标的工程项目规定》(发改委16号令) | 强制招标范围与规模标准 |
| 《标准施工招标文件》(2017版) | 投标文件格式规范（投标函、授权委托、工程量清单等标准格式） |
| 《政府采购法》及其实施条例 | 如涉及政府采购类项目 |
| 《电力建设工程施工安全监督管理办法》 | 安全生产条件、特种作业人员持证 |
| 《建筑业企业资质管理规定》 | 资质等级与承接范围 |

### 1.2 合规对 tender 设计的直接约束

**（一）投标有效期约束**

《招标投标法实施条例》§25 规定投标有效期从投标截止日起算。系统必须：
- 校验所有资质证书、人员证书、法定代表人授权委托书在"投标截止日 + 投标有效期"期间内不会过期
- 不是简单的"开标日"校验，而是"截止日 + 有效期（通常90天）"

**（二）签字盖章合规**

《电子招标投标办法》§27 要求电子投标文件须经电子签名（CA证书）。线下标书要求法定代表人（或授权代表）签字+单位公章。系统应：
- 自动检测各分册封面、投标函、承诺书是否标注签章位置
- 支持导出"签章位置总览图"，逐页标注需要签字/盖章处
- 对电子标（ECP平台），CA签章由平台完成，但 tender 应对齐ECP签章坐标

**（三）废标条款的法定边界**

《招标投标法实施条例》§51 规定了法定废标情形，系统应在符合性检查中区分：
- **法定废标**（必须检测）：无单位盖章、无法定代表人或授权代表签字、联合体无共同协议、报价超限价、工期超招标要求、质量不达标
- **约定废标**（按招标文件检测）：未按要求密封、正副本份数不符、业绩数量不足等
- **评标委员会自由裁量**（标记但不强制阻断）：技术方案不够详细、类似业绩关联度不够

**（四）投标文件保存期限**

《招标投标法》§35-36 及其实施条例要求评标资料保存≥15年。tender 归档策略应显式设定保存期限为≥15年，并支持导出独立可读的归档包（不依赖系统运行时环境）。

## 2. 国网公司招投标规则专项分析（重点）

### 2.1 国网ECP平台投标全流程

国网电子商务平台（ECP）是现代国网招投标的主要载体，tender 必须理解其流程：

```
ECP流程（当前spec/plan缺失）:
招标公告 → 下载招标文件（SGCC包/离线工具） → 
供应商资质能力核实（一纸证明） → 
线上提问/答疑/补遗 → 投标文件编制（离线工具） → 
投标文件加密/签名（CA） → 投标文件上传 → 
投标截止/解密开标 → 在线评标 → 
中标候选人公示 → 中标通知书（电子） → 
合同签订（ECP电子合同）
```

关键差异点：
1. **招标文件格式不是ZIP/PDF**，而是国网专用的 `.sgcc` 离线投标工具包，含结构化数据+PDF/DOCX附件
2. **投标递交不是打印装订**，而是CA加密后上传ECP，开标时在线解密
3. **开标不在现场**，而是在线直播，投标人远程查看开标价格
4. **澄清答疑全部在ECP平台内完成**，需解析ECP导出的答疑文件

### 2.2 国网供应商资质能力核实（一纸证明）

这是国网特有的制度，必须纳入系统设计：

- 国网每年对供应商进行资质能力核实（俗称"一纸证明"）
- 核实通过后获得《供应商资质能力核实证明》，有效期一年
- 持有有效"一纸证明"的项目，资格部分可以**简化响应**，直接引用核实证明
- 系统应在公司库中单独管理"一纸证明"状态，在投标时自动判断：
  - 如果一纸证明在有效期内且覆盖招标资质要求 → 简化资格文件，引用一纸证明编号
  - 如果一纸证明过期或未覆盖 → 按完整资格审查文件准备

> review 报告提到了资质许可证但未提及"一纸证明"制度，这是国网投标的核心效率优化点。

### 2.3 国网批次/框架协议采购

国网90%以上的招标采用"批次"方式（如"国网2026年第一批变电设备招标"），不是单项目招标。

**审慎评估：** 当前 spec 和 plan 中的"项目"模型是单项目单招，需判断是否扩展为批次模型。

建议的最小扩展方案（不改变核心架构，但预留）：

| 概念 | 当前模型 | 建议 |
|---|---|---|
| 批次（Batch） | 无 | project 表增加 `batch_id` 外键，可为 NULL；单项目时即为单项目 |
| 标段（Section） | 无 | project 表增加 `section_name` 字段（单项目或批次下标段） |
| 标包（Lot/Package） | 无 | project 表增加 `lot_name` 字段 |
| 框架协议（Framework Agreement） | 无 | project 表增加 `procurement_type` 枚举：`single` / `batch` / `framework` |

不建议首期做全批次建模，但**至少 project 表字段预留 batch_id、procurement_type、section_name、lot_name**，否则后续只能迁移不能扩展。

### 2.4 国网评标办法的专业维度

国网常用评标办法不只是"综合评估法/最低价法/合理低价法"三种大类，实际操作中：

- **综合评估法** 细分为技术评分+商务评分+价格评分，各自权重招标文件给定
- **经评审的最低投标价法** 并非纯最低价，而是经过技术/商务符合性评审后的最低报价
- **国网特有的基准价计算规则**：取所有有效投标报价的算术平均值（或去掉最高最低后的平均值）下浮一定比例作为基准价

> review 提到的 `evaluation_method` 字段粒度不足以支撑国网评标，建议 `evaluation_method` 拆为 `evaluation_method`（大类）+ `evaluation_detail`（JSONB：技术权重/商务权重/价格权重/基准价计算规则/下浮比例）

### 2.5 国网对人员社保的硬性要求

review 已提到社保校验，但国网的具体规则更严格：

- 国网通常要求项目经理、技术负责人、安全员等关键岗位"在本单位连续缴纳社保 ≥ 6个月"
- 有的项目要求"投标截止日前6个月连续缴纳"
- 有的项目采用"资格预审"，社保考核更早
- 需区分"社保在本单位"和"社保在分公司/子公司"（国网对分公司视为同一法人，子公司不算）

系统应在人员社保校验时支持配置"连续月数N"和"截止日期规则"。

## 3. 分包/标段/标包层级模型建议

### 3.1 当前设计的扁平淡化问题

当前 spec 和 plan 中的层级模型为：
```
项目 → 分册（商务标/技术标/报价标/资格审查册）→ 章节
```

但国网实际常见的层级为：
```
批次（如"2026年第一批变电设备"）
├── 标段1（如"500kV变压器"）
│   ├── 标包1（如"包A：某变电站#1主变"）
│   │   ├── 商务标
│   │   ├── 技术标
│   │   └── 报价标
│   └── 标包2
└── 标段2
```

### 3.2 建议方案

tender 首期不强制支持批次，但需做到：

1. **project 表新增字段**：`parent_project_id`（自引用外键，支持层级）、`procurement_type`、`section_name`、`lot_name`
2. **分册（volume）概念显式建模**：在 `bid_chapter` 之上增加 `bid_volume` 表，分册类型包括 `business`/`technical`/`pricing`/`qualification`/`attachments`
3. **分册与章节的关系外键化**：`bid_chapter.volume_id` 引用 `bid_volume.id`

这样单项目时退化为 1→N 分册→N 章节，批次时自然扩展。

## 4. 电子签章与加密递交（合规关键）

### 4.1 国网ECP平台CA签章机制

ECP平台使用国网统一CA数字证书（由中国金融认证中心CFCA或北京CA签发）：
- 商务文件（投标函、报价表）需要 CA 签章
- 技术文件（技术方案）需要 CA 签章
- 签章不是"在Word里插入一张图片"，而是国密SM2算法签名的数字信封

**tender 的定位**：不替代CA签章工具，但应：
- 导出符合ECP签章工具要求的中间文件
- 标注每个文件需要签章还是签字、是否需要骑缝章
- 生成"签章清单"供投标负责人逐项勾选确认

### 4.2 加密与递交时限

- ECP要求在投标截止前至少2小时完成上传（ECP系统公告明确建议）
- 加密操作在本地完成（国网离线投标工具）
- 上传后不能撤回或修改

spec §11 已有截止时间72h/24h/2h预警，这是好的，但需补充：
- **ECP平台公告的"建议上传时间"追踪**（每批次ECP都会发布上传窗口公告）
- **投标文件大小限制检测**（ECP对单个文件有大小限制，通常商务文件<50MB，技术文件<200MB）
- **文件格式白名单**（ECP只接受特定格式：PDF、DOCX、XLSX、JPG、PNG等，ZIP格式可能被拦截）

## 5. 绿色数智化评标趋势与数据准备

### 5.1 国网绿色供应链评价

国网自2023年起全面推行绿色数智评标，招标文件中出现以下新要求：

- **绿色企业评价**：碳排放、绿电使用比例、绿色工厂认证
- **绿色产品/服务**：低碳施工方案、节能设备选用、绿色建材占比
- **数智化能力**：BIM、数字化管理平台、智慧工地
- **碳足迹报告**：部分试点项目已要求提供

当前 master_data 的 evidence taxonomy 已包含 `green_development_document`、`green_management_system`、`esg_document`、`green_power_certificate`，这是好的。但 spec 的技术标章节未体现绿色施工方案的独立要求。

### 5.2 建议

- 技术标 §4 章节增加"绿色施工及低碳管理措施"（建议放在质量/安全章节之后）
- 公司库增加"绿色企业/绿色产品/数智化"标签体系
- 评分项提取时显式标注"绿色评分项"

## 6. spec 层面的遗漏与修正建议

### 6.1 资格审查方式差异未展开

spec §2.1 项目立项字段中 review 补了 `qualification_review_type`（资格预审/资格后审），但未展开两者差异对系统的影响：

| 差异维度 | 资格预审 | 资格后审 |
|---|---|---|
| 文件产出时机 | 投标前单独提交资格预审申请文件 | 随投标文件一同提交 |
| 审查内容 | 仅资质/业绩/人员，不涉及技术方案和报价 | 全部内容 |
| 有效期 | 通过后在招标批次内有效 | 仅当次投标有效 |
| tender 工作流差异 | 需支持"预审文件生成"独立流程 | 资格审查册作为投标文件分册之一 |

建议在工作流状态枚举中区分 `pre_qualification_drafting` → `pre_qualification_submitted` → `pre_qualification_passed/failed`。

### 6.2 答疑/补遗/澄清的生命周期管理

review 已提到答疑/补遗合并，但缺少对以下场景的处理：

- **多轮答疑**：招标人可能发布多轮答疑（通常2-3轮），每轮答疑可能修改前一轮的结论
- **延期通知**：答疑阶段招标人可能同时发布开标延期通知，系统应自动更新所有时间依赖的校验
- **补遗内容与原始招标文件的冲突**：补遗效力 > 原招标文件，系统需提示覆盖区域

建议在 plan Phase 3 增加：
- 答疑/补遗版本链（clarification_round 1→2→3）
- 覆盖影响分析（"补遗#2 修改了第三章评分办法，以下章节需重新生成：3.1、3.2、4.5"）

### 6.3 联合体投标场景

review 中 `is_consortium_allowed` 字段有了，但未展开联合体投标对系统的完整影响：

- **联合体协议书**生成（法定格式，各成员签字盖章）
- **各成员资质/业绩/人员的分别管理**：系统需支持关联多个公司库中的主体
- **联合体牵头人指定**：需标记哪个成员为牵头人（投标函由牵头人出具）
- **联合体各方的份额/分工说明**（技术标需要）

当前公司库是单主体（library_company），联合体场景下需支持"从多个 library_company 中各取所需资质/人员/业绩"。

## 7. plan 层面的遗漏与修正建议

### 7.1 执行顺序调整建议

plan §Suggested Execution Order 将 Phase 7（图表）排在 Phase 9 之前，review 认为图表可以更早。但从投标实务看，**组织架构图是技术标的核心章节**，缺少图表则技术标不完整，建议：

调整后顺序：
```
1→2→3→4→5→6→7（图表提前至6之后，与6并行）→8-pre（符合性检查）→8→9→10→11→12
```

理由：技术标编写（Phase 6）与图表生成（Phase 7）天然耦合——技术标正文引用图表，图表规格来源于技术标内容。两者应同时或紧接开发。

### 7.2 测试矩阵应补的国网专项用例

review Phase 12 提到"4类业务线各一组端到端fixture"，这是好的，但需增加：

- [ ] 国网ECP导出模拟：生成的文件结构是否对齐ECP离线工具要求
- [ ] 法定废标条款检测：逐条验证符合性检查是否100%命中法定废标
- [ ] 一纸证明有效期分支：有效期内的简化路径 vs 过期的完整路径
- [ ] 联合体协议生成：多主体资质/业绩能从候选公司库正确筛选
- [ ] 答疑多轮覆盖：3轮答疑→检测最后一轮覆盖是否正确
- [ ] 投标有效期证书过期检测：投标截止日+90天后的过期预警

### 7.3 数据库迁移风险

plan Phase 1 要求在 project 表加大量字段，但当前 project 表仅有 `id`、`name`、`created_at` 三个字段。需注意：

- 新增字段应采用 `ADD COLUMN IF NOT EXISTS` + `DEFAULT` 避免锁表
- 工作流状态从无到有的迁移策略：存量项目统一设为 `drafting`（假设已进入生成阶段）或 `constraints_pending_confirmation`
- project 表结构变更影响所有 downstream API（现有 `ProjectOut` 仅含 id+name），需要前后端同步升级

## 8. 对 review 报告的勘误与补充

review 报告整体质量高，以下为少量勘误和补充：

### 8.1 勘误

1. **§3.5.3 业绩四件套**：review 描述为"中标通知书 + 施工合同 + 竣工验收证明 + 业主出具的履约证明"。国网实务中更准确的是"中标通知书 + 施工合同 + 竣工验收报告/竣工验收证明书 + 业主评价/履约证明"。其中"业主评价"并非所有项目都要求，有些项目只需前三件。建议系统设计为"3+1"模式（3件必须 + 第4件按招标要求）。

2. **§3.7 网络计划图**：review 建议用双代号/单代号网络计划图，这在大型变电站项目中确实需要。但首期建议仅支持**双代号时标网络计划图**（最常用），单代号留 Phase 7-bis。

### 8.2 遗漏项补充

review 遗漏了以下必须考虑的事项：

| 遗漏项 | 重要性 | 建议 |
|---|---|---|

### 8.3 review 的优先级分配调整建议

review §8 的优先级分配基本合理，但以下几项应升级：

- **一纸证明管理** 应从 P2 升级至 P0（国网投标的核心效率要素）
- **电子签章清单导出** 应从 P1 升级至 P0（缺签章=废标，符合性检查必须覆盖）
- **投标有效期叠加逻辑** 应从 P2 升级至 P0（证书校验逻辑的错误会导致废标）
- **联合体协议生成** 保持 P1（首期可限定单主体投标）
- **批次/框架预留** 从无提到 P1（字段预留成本极低，不预留未来改造成本极高）

### 9.2 review 建议与 spec/plan 的冲突检测

- review 建议 `delivered` → `submitted`，但 spec 中使用 `delivered` 作为交付完成状态。如果交付=递交，确实应改名；如果交付=打包完成待递交，则应保留。**建议澄清：交付=文件生成完毕并导出；递交=已送到/已上传。两个状态都应存在。**

- review 新增 Phase 8-pre "符合性检查"独立于 Phase 8（AI审查），这是正确的。但需注意：符合性检查应是**确定性规则引擎**（可编程规则，不依赖AI），AI审查应是**非确定性辅助**。

## 11. 特别提示

### 11.1 关键风险——当前 project 表太薄

当前 `project` 表仅有 `id`、`name`、`created_at` 三个字段，而修订后的设计需要30+字段。这不只是"加字段"的问题，而是涉及：

- 前后端所有接口返回的 project 对象都要扩展
- 现有项目数据迁移策略（存量项目的字段默认值）
- 前端项目创建工作台需全新设计

**建议 Phase 1 作为整个改造的技术基础先行，且需要完整的数据库迁移方案评审。**

### 11.2 关键风险——.sgcc格式解析

国网离线投标工具生成的 `.sgcc` 格式是私有二进制格式，非公开标准。如果 tender 无法直接解析（大概率如此），需要确定替代方案：

- 方案A：解析招标文件导出的 PDF/DOCX 版本（国网通常同时提供可打印版本）
- 方案B：用户手工将关键信息录入系统
- 方案C：OCR/文本提取国网招标文件PDF

建议首期采用方案A+B组合，方案C作为辅助。spec 需明确声明"不直接解析.sgcc二进制格式，以招标文件PDF/DOCX版本为输入"。

### 11.3 关键风险——外部报价文件挂载的边界

review 明确提出"经济标不在首期范围内，仅支持外部文件挂载"。但挂载边界需明确：
- 是否支持外部报价文件的**页面插入**？（如报价表插入到商务标特定位置）
- 是否支持**目录包含**外部文件的页码？（目录需要知道报价文件的页码）
- 是否支持**打包/合并**为一个PDF？（外部PDF与内部DOCX合并的技术复杂度）

建议 spec §7.2 增加一个简短的"外部文件挂载策略说明"。

**如果只能做5件事，我建议做这5件:**

这5件事做好，首期即可用于真实国网投标辅助（人工复核+系统检查），而非仅是"AI写稿工具"。
