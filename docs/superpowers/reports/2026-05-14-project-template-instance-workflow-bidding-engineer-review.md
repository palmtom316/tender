# 项目模板实例工作流计划评审报告（施工单位投标工程师视角）

**评审日期：** 2026-05-14
**被评审文档：** [`docs/superpowers/plans/2026-05-14-project-template-instance-workflow.md`](../plans/2026-05-14-project-template-instance-workflow.md)
**评审视角：** 施工单位投标工程师（一线作业人员）
**评审结论：** 工程架构可行，业务覆盖不足；建议至少补足 P0 后再开工

---

## 一、总体评价

计划在工程层面是**过关**的：

- 三层结构合理（全局模板 / 项目模板实例 / 生成内容），避免污染基础模板。
- 引入"模板调整工作台"对接招标文件解析的目录差异融合，正中投标工程师最大痛点之一。
- 表单化模板维护界面，避免技术门槛。
- 修订快照（revision snapshots）便于追溯。
- 章节级状态管理，便于多人分工。
- 审查问题回路按源头分类（模板 vs 内容）。

但从一线投标工程师的实际作业场景看，**计划仍偏「模板管理系统」而非「投标系统」**。投标日常作业里的**答疑合并、盖章签字、条款响应、报价、资质过期、协作锁、截止时间**这些核心场景没有进入数据模型与任务列表。这些不是体验问题——任何一项缺失都可能导致**实质性废标**或一线绕开系统用 Word 兜底。

---

## 二、P0 级问题：必须解决，缺一项就有废标风险

### P0-1 缺「答疑 / 补遗 / 澄清」的二次合并流程

**问题描述：** 招标过程中，招标人会发布**答疑文件 / 补遗 / 招标文件澄清**，频繁修改评分办法、技术参数、提交时间、目录要求、格式要求。这是投标周期里**最高频也最危险的变更源**。

**计划现状：** 解析只发生一次（`upload → parse → requirements → template`），没有"补遗到达后的二次解析—差异融合—模板再调整"路径。

**建议：**
- `tender_document` 表加 `document_type`（招标书 / 答疑 / 补遗 / 中标候选人公示）。
- `reconcile-directory` 支持**增量比对**（基于上次确认快照）。
- 答疑触发的差异以"必须处理"高优先级显示，禁止跳过。

---

### P0-2 盖章签字位没有独立块类型

**问题描述：** 招标人对盖章签字位置极其严格："投标函封面 / 末尾盖单位公章 + 法定代表人签字"、"商务标每页骑缝章"、"资格审查表逐页签字"。

**计划现状：** 模板里如果只是 `fixed_text` 写一句"此处盖章"，预览看不出来，最终人工漏盖直接废标。

**建议：**
- 块类型新增 `seal_mark`，支持子类型：公章 / 法人章 / 骑缝章 / 法人签字 / 项目经理签字。
- `TemplatePreviewPane` 用显眼标记渲染所有盖章位。
- 增加"提交前盖章清单逐项勾选"硬性确认流程。

---

### P0-3 格式规则深度不够

**问题描述：** 实际招标常硬性规定：
- 正文字体（"宋体小四，行距固定值 22 磅"）
- 页边距、段落缩进
- 封面 / 投标函样式
- 装订（胶装 / 骑马钉 / 双面打印 / 单独密封报价）
- 目录页层级（"自动生成三级目录，与正文页码一致"）

**计划现状：** 计划提了 `formatting_rule` 作为审查源，但模板块类型只有 `page_break` + `header_footer`。`docx_renderer` 不实现这些，导出后必须人工 Word 调，效率被吃光。

**建议：**
- 在 `project_template_instance.metadata_json` 下加 `format_profile`（字体、间距、页边距、装订）。
- 增加 `seal_unit` 概念（密封分册：商务标 / 技术标 / 资格审查 / 报价单独装订）。

---

### P0-4 条款级响应对齐缺失（响应/偏离表）

**问题描述：** 这是投标工程师**每个项目都要做的核心动作**：把招标文件每一条技术/商务条款，对应填写"完全响应 / 部分响应 / 偏离"。实质性响应不到位会直接评分扣光或被废标。

**计划现状：** 只做了**目录级**对齐（chapter ↔ tender directory），没做**条款级**对齐（requirement ↔ response）。

**建议：**
- 在 `project_template_chapter` 之外引入 `requirement_response` 表。
- 把已解析的 `bid_requirements` 与模板的 `block / chapter` 双向绑定。
- 生成时强制每条要求至少有一个回应块。

## 三、P1 级问题：强烈建议补，否则一线会绕开系统

### P1-1 业绩 / 资质 / 人员证书库与"过期校验"

招标 90% 都要求"近 X 年类似项目业绩"、"人员资格证书在有效期内"、"安全生产许可证有效"。

- 当前 `asset_placeholder` 是泛化占位符，没有"业绩筛选条件"、"证书有效期校验"。
- 建议：单独的资质 / 业绩 / 人员模块，`asset_placeholder` 增加 `asset_filter`（项目类型、合同金额、完工时间）和 `valid_until_field`，预览时过期素材红色告警。

---

### P1-2 报价 / 工程量清单（BOQ）结构化不见

商务标核心是**报价表 / 工程量清单 / 单价分析表 / 措施项目费 / 暂列金额**，全是结构化数据 + 公式（合价 = 单价 × 数量）。

- 计划里块类型清一色文本 / AI / 占位符，承不住报价。
- 建议：单独 `pricing_block` / `excel_attachment` 类型，或明确承认"报价独立模块"另开计划。**但模板调整工作台对接报价模块的边界必须现在画清楚**，否则后期没法收口。

---

### P1-3 多人协作 / 章节锁

商务标和技术标通常并行准备，2-5 人同时编辑同一份标书很常见。

- 当前计划没有 `chapter_lock` / `currently_edited_by` / 乐观锁版本号，多人编辑必冲突。
- 建议：`project_template_chapter` 加 `lock_owner` / `locked_until`，前端章节树显示在编辑者头像。

---

### P1-4 投标截止时间硬约束

投标系统的本能：在截止时间前自动锁定最终版本、不允许再修改、保留快照。

- 计划有 `locked_for_generation` 状态，但没和**项目 `bid_deadline` / 提交动作**联动。
- 建议：`project` 表必须有 `bid_deadline`，到期前 N 小时自动提示"是否锁定最终版本"，提交后只读，便于评标质疑期回溯。

---

## 四、P2 级问题：体验 / 效率，影响留存但不影响中标

### P2-1 多方案 / 变体管理

大项目常并行准备「技术领跑型 / 性价比型 / 跟标型」多份方案。默认 1 个 project → 1 个 instance 会逼工程师在系统外另存多份 Word。

- 建议：`project_template_instance` 加 `variant_label` + `is_primary`，允许同 project 多个 instance。

---

### P2-2 反向沉淀路径（从已有标书学习）

投标公司一般都有几百份历史投标。计划方向是"全局模板 → 项目实例"，缺反向沉淀。

- 现有 `template_promotion_proposal` 是个好基础，但触发动作是"项目结束时手动提议"。
- 建议补一个入口"从历史标书（DOCX）反向解析为候选模板"。

---

### P2-3 多级会签 / 评审流

正式投标前通常技术总工 → 商务经理 → 法务 → 分管副总逐级会签。

- 计划只有"模板提议合并"的评审流，没有**标书本身的提交前评审流**。
- 建议：单独 `bid_review_flow` 表，定义会签节点和签字记录。和 P0-2 有重叠。

---

### P2-4 变量字典需固化

`variable` 块过于抽象。投标的核心变量集是固定的：项目名称、招标编号、招标人、投标人、法定代表人、项目经理、工期、报价。

- 建议：内置一套**项目级标准变量**（`project_variables` JSONB 或表），首次解析招标文件后自动填充，模板里直接引用 `{{project.tender_no}}` 而不是让工程师每次自定义变量名。

---

## 五、任务粒度上的具体建议

| 任务 | 问题 | 建议 |
|---|---|---|
| Task 4 | reconciliation 测试用例偏轻，六种动作只覆盖了**章节级**。 | 单元测试需包含**小节级**（X.Y.Z 三级）嵌套与跨层级移动。 |
| Task 7 | 生成阶段缺性能基线。典型技术标 300-500 页，章节几十个。 | 加一条"生成 200+ 章节实例的耗时上限"，避免上线才发现卡死。 |
| Task 8 | 现有 6 个 `issue_source` 不全。 | 补 `tender_addendum`（答疑导致）、`seal_missing`（盖章漏项）、`requirement_not_responded`（条款未响应）三类。 |

## 六、优先级建议

如果是 **MVP 交付**：

- **必须纳入：** P0-1（答疑/补遗）、P0-2（盖章块）、P0-4（条款级响应）。
- **可占位下期：** P0-3（格式规则）放进 `metadata_json` 占位，下一版本再做。
- **暂缓：** P1 / P2 全部。

如果是 **完整版交付**：

- **第一阶段（4 周）：** P0 全部 + P1-3（协作锁）+ P1-4（截止时间）。
- **第二阶段（4 周）：** P1-1（资质/业绩库）+ P1-2（报价模块边界）。
- **第三阶段（按需）：** P2 全部。

---

## 七、一句话总结

**计划的工程架构是过关的，但产品定位偏「模板系统」而不是「投标系统」**。把"全局模板 → 项目实例 → 生成内容"这条线走通了，却没把投标工程师日常作业里的**答疑合并、盖章签字、条款响应、报价、资质过期、协作锁、截止时间**这些核心场景纳入。如果不补 P0，一线投标工程师拿到这个系统会发现**还得用 Word 兜底**，反而比纯 Word 多一层流程负担。
