# 前端工作台重构设计

**日期：** 2026-03-17  
**范围：** `frontend/` 内部控制台界面重构，仅涉及前端信息架构、视觉系统与交互层，不改动后端业务语义。  
**目标：** 将当前按流程分散的页面重构为统一工作台，形成“左侧模块导航 + 右侧卡片化工作区 + 顶部滚动提示 + 右上角 Copilot 助手”的浅色极简界面。

---

## 1. 背景与当前问题

当前前端以单页直出和独立页面为主：

- `frontend/src/App.tsx` 通过路径判断直接切换页面；
- `frontend/src/styles.css` 以单一全局样式为主，缺少统一设计 token；
- 各业务页面存在较多局部样式和信息密度不一致的问题；
- 现有页面更像“流程页”，不符合“工作台”式产品体验。

这会带来三类问题：

1. 缺少稳定壳层，模块切换、设置、用户入口无法统一；
2. 视觉与交互不一致，按钮、卡片、提示区样式分散；
3. 新模块（如投标资料库多标签、Copilot 助手）难以平滑接入。

---

## 2. 设计目标

本次重构聚焦以下目标：

- 建立统一工作台壳层，替代当前“每页独立头部”的结构；
- 采用 `minimalist` 浅色风格，整体有质感、层级清晰、无冗余说明文字；
- 左侧导航区域颜色略深于右侧工作区；
- 左侧按钮具备轻微“粘土按压感”，但不夸张、不影响极简感；
- 同一模块内的不同工作通过顶部标签切换；
- 工作区内容统一以卡片方式呈现，状态信息内聚到卡片；
- 顶部提供横向滚动的提示/警示条；
- 右上侧提供平时隐藏的 `Copilot` 助手，用于人机交付协作；
- 所有工作区按钮风格、大小、颜色、描边、阴影完全统一。

---

## 3. 目标信息架构

### 3.1 一级模块

批准后的一级模块结构如下：

1. `项目总览`
2. `投标资料库`
3. `要求确认`
4. `章节编辑`
5. `审校导出`
6. `设置`

### 3.2 左侧导航区

左侧导航为固定侧栏，包含：

- 顶部：品牌区 + 汉堡收缩按钮；
- 中部：一级模块按钮列表；
- 设置：作为一级模块按钮之一；
- 底部：用户名、当前角色、用户管理圆形按钮。

收缩态要求：

- 仅保留图标与必要状态；
- 汉堡按钮始终可见；
- 用户管理圆形按钮保留；
- 不影响当前模块高亮与键盘访问。

### 3.3 右侧工作区

右侧为统一内容工作区，包含：

- 顶部滚动提示条；
- 模块标题区；
- 模块级标签页；
- 卡片网格 / 卡片列表内容区；
- 右上侧 `Copilot` 隐藏式助手入口。

### 3.4 投标资料库模块标签

“投标资料库”模块固定包含以下标签页：

1. `历史投标文件`
2. `优秀投标文件`
3. `规范规程`
4. `公司资料`
5. `人员资料`

各标签页内容采用统一卡片结构承载文件、状态、标签、更新时间、操作按钮。

---

## 4. 视觉系统

### 4.1 总体风格

视觉方向为：

- 浅色；
- 极简；
- 有质感；
- 柔和边界；
- 克制的层级阴影；
- 接近高端工作台产品，而非营销页。

避免：

- 大面积高饱和色；
- 过度渐变；
- 厚重拟物；
- 大段说明文字；
- 不统一按钮尺寸；
- 卡片风格混杂。

### 4.2 色彩策略

建议使用语义 token，而不是组件内硬编码颜色。

- 左侧侧栏：比主内容区略深的暖灰/冷灰浅色；
- 主工作区：更明亮的米白或冷白；
- 主强调色：低饱和蓝绿或灰蓝，用于选中态与关键按钮；
- 风险态：红/橙只用于提示条与风险卡片，不扩大使用范围；
- 边框：超低对比细描边；
- 阴影：短距离柔和阴影，强调材质感。

### 4.3 按钮系统

工作区按钮必须统一：

- 同一高度；
- 同一圆角；
- 同一边框；
- 同一阴影；
- 同一字体大小与字重；
- 不因场景改变而出现“另一个按钮体系”。

左侧导航按钮额外具备轻微粘土效果：

- 默认态：轻柔外凸；
- Hover：阴影稍增强；
- Press：轻微内陷/下压；
- Active：保留清晰选中底色与描边。

### 4.4 卡片系统

卡片为工作区唯一主承载体：

- 文件卡；
- 状态卡；
- 审校问题卡；
- 统计卡；
- 设置项卡；
- 空状态卡。

卡片规则：

- 统一背景、圆角、描边、阴影；
- 标题与状态在首屏即可判断；
- 操作按钮不超过 2 个主操作；
- 文本行数尽量克制；
- 支持多列网格与单列列表两种布局。

---

## 5. 交互设计

### 5.1 顶部滚动提示条

顶部提示条用于呈现系统提示、警示、处理中状态，不承载说明文档性质内容。

要求：

- 横向文字滚动；
- 可根据状态切换颜色；
- 高度小巧，不抢占主内容；
- 支持无内容时折叠；
- 可放置解析中、未确认否决项、阻断问题等提醒。

### 5.2 模块内标签切换

同一模块内的工作切换统一采用顶部标签页。

要求：

- 视觉清晰但不过度厚重；
- 支持数量较多时横向滚动；
- 当前标签高亮；
- 切换后保留工作区壳层不闪烁；
- 可与浏览器路径或查询参数同步。

### 5.3 Copilot 助手

`Copilot` 位于右上侧边缘，默认隐藏。

触发方式：

- 鼠标经过时展开；
- 快捷键 `Ctrl/Cmd + K` 唤起；
- 点击外部区域后收起。

作用：

- 作为人机协同面板；
- 展示协作建议、待办、下一步动作；
- 不阻塞主业务区使用。

### 5.4 设置区

设置作为一级模块，主要用于 AI 模型 API 填写。

设置区信息组织：

- 模型服务卡；
- API Key / Base URL / 模型名输入项；
- 连通性状态卡；
- 保存按钮；
- 安全提示与错误状态。

尽量减少说明文字，优先用字段标题、状态标签和卡片层级表达含义。

---

## 6. 页面映射与迁移原则

### 6.1 当前页面到新工作台映射

- `project-list.tsx` → `项目总览`
- `upload.tsx` + `parse-results.tsx` → `投标资料库` 的相关标签/卡片
- `requirements-confirmation.tsx` → `要求确认`
- `chapter-editor.tsx` → `章节编辑`
- `review-results.tsx` + `export.tsx` → `审校导出`

### 6.2 迁移原则

- 优先引入统一壳层，不先深改业务逻辑；
- 保留现有数据请求与 API 语义，先改呈现方式；
- 将内联样式迁移到统一 token 与组件样式；
- 对同类状态（空、错、加载、阻断）形成统一组件表达；
- 尽量减少页面级重复头部和返回链接。

---

## 7. 技术设计

### 7.1 建议组件层级

建议新增或重组以下前端单元：

- `layout/`：工作台壳层、侧栏、顶部条、内容容器；
- `components/ui/`：按钮、卡片、标签、滚动提示、状态徽标；
- `components/workspace/`：模块头、标签条、空态、统计卡；
- `pages/`：保留业务页，但由“完整页面”改为“工作区内容片段”。

### 7.2 样式组织

建议从单一 `styles.css` 逐步拆分为：

- 全局 token；
- 布局样式；
- 通用组件样式；
- 页面/模块样式。

若当前阶段不引入 CSS-in-JS 或 Tailwind，则继续使用 CSS 文件，但必须 token 化。

### 7.3 路由策略

短期内可继续使用当前路径解析方式，但应抽象：

- 一级模块路径；
- 模块内标签状态；
- 选中项目上下文。

后续若切到 TanStack Router，可在不推翻壳层的前提下迁移。

---

## 8. 可用性与一致性要求

必须满足：

- 左侧与右侧背景区分明确；
- 所有按钮风格完全一致；
- 可点击元素具备清晰 hover / active / focus 状态；
- 左侧收缩后仍可用；
- 重要状态在顶部提示条与卡片中双重可感知；
- 卡片布局在中等屏幕下保持整洁；
- 不依赖 hover 作为唯一触发方式；
- `Copilot` 快捷键可触达；
- 表意尽量通过结构和状态，而不是长文说明。

---

## 9. 验收标准

设计验收以以下结果为准：

1. 前端具备统一工作台壳层；
2. 左侧侧栏支持展开/收缩；
3. 左侧底部具备用户名与用户管理圆形按钮；
4. 顶部存在滚动提示条；
5. “投标资料库”具备 5 个固定标签；
6. 工作区内容以卡片为主；
7. 工作区按钮视觉完全统一；
8. 右上存在默认隐藏、可悬停或快捷键唤起的 `Copilot`；
9. 页面整体为浅色极简风，左侧略深、右侧略浅；
10. 无非必要说明文字。

---

## 10. 风险与边界

本次重构不包含：

- 后端接口语义重写；
- 新业务流程设计；
- 权限体系深度重构；
- Copilot 后端智能能力实现。

主要风险：

- 当前页面逻辑与布局耦合较深，迁移时需先抽离壳层；
- 无前端测试基线，建议先补最小 UI 验证能力；
- 单文件全局样式过大，若不拆分会继续增加维护成本。

---

## 11. 结论

推荐采用“先壳层、后模块迁移、再统一交互细节”的渐进式重构路径。  
优先收益点是统一布局、模块导航、标签式工作区与卡片化呈现；在不大改业务 API 的前提下，可显著提升产品完成度、专业感与后续扩展性。
