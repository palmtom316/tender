# 前端旗舰级 UI 统一与优化设计

**日期**：2026-05-12  
**范围**：`frontend/` 全项目页面、共享 UI 组件、CSS token 与模块级样式。  
**目标**：将现有投标工作台统一为专业稳重、低干扰、高一致性的企业级生产界面；统一按钮、标签、卡片、表格、表单、状态与页面布局。

## 1. 背景与根因

项目已具备 `AppShell`、`Sidebar`、`WorkspaceTabs`、`ClayButton`、`Badge`、`Card` 与分层 CSS token，但页面仍存在风格漂移：

- 多个模块直接使用原生 `<button>` 和局部 class；
- `asset-tabs`、`filter-chip`、`template-preview-tabs` 等局部 tab/pill 重复造轮子；
- 表格存在 `data-table`、`asset-table`、模块私有表格等多套密度和操作列规则；
- 局部内联 style 破坏 token 化和主题一致性；
- Badge/Tag 语义在状态、分类、风险之间混用；
- 页面重排程度不一，导致用户在不同模块中需要重新学习操作区位置和状态含义。

根因不是颜色不够好看，而是**设计系统没有成为页面实现的唯一来源**。因此必须先归一化基础组件，再进行页面级旗舰化。

## 2. 已批准设计上下文

### 用户
主要用户是投标专员 / 商务技术人员。他们长时间进行上传、解析、要求确认、资料引用、章节生成、审校和导出。

### 品牌气质
专业、稳重、可信。界面应增强掌控感与工作效率，不追求炫技。

### 质量目标
旗舰级重做：不仅统一视觉，还要逐页优化信息层级、状态反馈、空状态、操作密度与响应式细节。

## 3. 设计原则

1. **一致性优先于局部惊艳**：所有页面共享 Button、Badge/Tag、Card、Tab、Table、Form、Toolbar 规则。
2. **高密度但低干扰**：减少重复说明，突出风险、状态、下一步。
3. **状态必须可解释**：每个失败、等待、成功、阻断状态都给出清晰含义和恢复动作。
4. **主操作克制**：每个区域最多一个主按钮；批量/次级动作使用 outline/ghost。
5. **渐进实施**：先设计系统，再核心页面，再剩余页面 polish，降低回归风险。

## 4. 统一视觉系统

### 4.1 色彩

保留当前暖棕主色，但收敛使用范围：

- `primary`：关键动作、当前选中、核心强调；
- `info`：解析/处理中/中性提示；
- `success`：已完成/已确认/可导出；
- `warning`：需关注/临期/待补充；
- `danger`：失败/阻断/删除/高风险；
- `neutral`：分类、来源、普通标签。

禁止在模块内硬编码临时颜色。新增颜色必须先进入 token，并说明语义。

### 4.2 字体与密度

- 主体字号保持 13–14px，适配长时间阅读和高密度表格；
- 标题层级减少到页面标题、区块标题、卡片标题三级；
- 操作按钮、标签、表格文字使用统一字号和字重；
- 避免营销式大标题和大段说明。

### 4.3 材质与阴影

- 卡片使用统一浅色 surface、细边框、轻阴影；
- clay 质感只用于按钮和可点击导航，不扩大到所有容器；
- hover/active/focus 状态必须明确但克制。

## 5. 组件系统设计

### 5.1 Button

统一 `ClayButton` 为全项目唯一按钮入口。

变体：

- `primary`：关键提交/生成/保存/导出；
- `outline`：普通操作；
- `ghost`：低权重操作、工具栏操作；
- `danger`：删除、清空、退出、不可逆操作；
- `secondary` 保留兼容，但映射到 `outline`。

要求：

- 高度、圆角、字体、间距统一；
- loading/disabled/focus 状态完整；
- 图标按钮使用统一尺寸；
- 模块内原生 `<button>` 要么替换为 `ClayButton`，要么使用共享 tab/chip/button class。

### 5.2 Badge / Tag

统一状态、风险、分类标签。

- `Badge` 表示状态结果；
- `Tag`/轻量 pill 表示分类、来源、筛选条件；
- 风险等级必须固定语义色：阻断=danger，冲突=warning，已确认=success，处理中=info。

### 5.3 Tabs / Chips

统一 Workspace 顶部 tab、资产分类 tab、要求筛选 chip、模板预览 tab 的视觉语言：

- tab 用于互斥视图切换；
- chip 用于筛选条件；
- 必须具备 `role=tab` 或按钮语义、当前态、键盘 focus。

### 5.4 Table

建立统一表格规则：

- 表头低对比背景；
- 行高适中，支持高密度扫描；
- 操作列右侧固定视觉风格；
- 空状态不留白，给出下一步；
- 附件链接、状态列、日期列、风险列统一呈现；
- `data-table` 与 `asset-table` 应共享基础 table token。

### 5.5 Forms

统一输入框、选择框、文本域、帮助文本、错误文本：

- label 在上，帮助文本在下；
- 错误态使用 danger 边框和短句说明；
- 表单操作区固定在卡片底部或 drawer 底部；
- 禁止散落的内联 margin/padding。

### 5.6 States

所有异步区域统一：

- loading：骨架或 compact loading，不只显示纯文本；
- empty：说明当前为空、为什么为空、下一步是什么；
- error：错误原因 + 重试/返回/联系管理员路径；
- success：短反馈，不打断工作流。

## 6. 页面级旗舰化范围

优先级从业务压力和风格漂移程度决定。

### P1：要求确认

目标：成为高密度风险处理工作台。

- 风险卡、筛选 chip、要求列表、批量确认操作统一；
- 当前筛选、阻断/冲突/待确认状态一眼可见；
- 原文查看、确认、忽略等动作位置固定；
- 减少重复描述，强化下一步。

### P1：章节编辑

目标：降低长时间编辑疲劳。

- 编辑区、生成状态、原文/要求引用区层级清晰；
- 章节操作区主次明确；
- AI 生成/重试/保存/导出状态统一。

### P1：投标资料库

目标：统一资料、人员、资产、规范、模板字段的复杂管理界面。

- 资产/personnel/company/template/standard 子页面共享 tab、table、toolbar、drawer、empty state；
- 附件、有效期、资质状态、来源标签统一；
- 移除局部自定义 pill/button 样式漂移。

### P2：项目总览

目标：清楚表达项目进度和下一步。

- 项目卡和统计卡统一；
- 最近项目、待处理风险、下一步动作结构化；
- 入口页不堆叠说明文案。

### P2：审校导出

目标：统一问题、合规、导出历史。

- 审校问题卡和合规表统一状态；
- 导出可用性、阻断项、历史记录清楚；
- 导出主按钮只在条件满足时突出。

### P3：设置

目标：表单和用户管理统一。

- AI 模型配置、API Key、用户管理、连接状态共享表单和状态规则；
- 删除内联 style；
- 危险动作显式化。

## 7. 响应式与可访问性

- 移动端不隐藏关键功能，只改变布局；
- 所有交互元素最小触控目标 44px，紧凑表格内例外需有可点击区域补偿；
- focus-visible 不得移除；
- 状态不能只靠颜色表达；
- 支持 `prefers-reduced-motion`；
- 保持浅色/暗色 token 可用。

## 8. 实施边界

本次只改前端表现层和必要组件抽象：

- 不改变后端 API 语义；
- 不改变核心业务流程；
- 不引入大型 UI 框架；
- 不做新功能，只改善一致性、布局、状态、可访问性和可维护性；
- 不为单一页面创建新的一次性视觉体系。

## 9. 验收标准

1. 全项目主操作按钮均通过 `ClayButton` 或共享按钮 class 表达；
2. Badge/Tag 状态语义统一，无页面私有状态色；
3. 主要 tab/chip 视觉一致，当前态和 keyboard focus 明确；
4. `data-table` / `asset-table` 等表格共享基础密度、表头、操作列规则；
5. 主要页面无影响布局的内联 style；
6. 要求确认、章节编辑、投标资料库三大核心页面完成页面级重排；
7. build、test 通过；
8. 页面在浅色主题下专业稳重、低干扰、高一致；暗色主题不破版。

## 10. 推荐实施阶段

### Phase 1：设计系统归一化

更新 token、按钮、标签、tab/chip、table、form、state 样式和组件入口。

### Phase 2：三大重页旗舰化

优先处理要求确认、章节编辑、投标资料库，修复最大风格漂移和最高频工作流。

### Phase 3：剩余页面 polish

项目总览、审校导出、设置、通用 modal/drawer、Copilot、错误边界。

### Phase 4：验证与清理

运行测试、构建，搜索原生按钮/内联 style/私有 badge/tab/table 残留，必要时补充 UI smoke test。
