# 投标文件生成流程改造设计

**日期:** 2026-05-07

**目标:** 将当前系统从“招标文件解析 + 大纲/章节生成 + 交付包”的基础能力，升级为一条可确认、可追踪、可审查、可归档复盘的投标文件生产流水线。

---

## 1. 结论先行

建议把投标文件生成拆成三类能力，而不是让 AI 一次性写完整标书：

1. **约束工程:** 招标文件解析结果、规范文件、评分办法、模板要求、用户确认意见统一沉淀为“投标编写约束”。所有生成、审查、排版都必须引用这些约束。
2. **结构工程:** 根据投标类型自动匹配模板包，再用解析出的章节要求、评分项、格式要求对模板大纲做差异融合，形成用户确认的大纲。未确认大纲不得进入正文生成。
3. **内容工程:** 商务标以公司资料和证明文件的确定性插入为主，AI 只辅助摘要、响应说明、缺失提示；技术标以“逐章计划、逐章生成、逐章审查、逐章修订”为主，并保留需求来源和评分点映射。

这条路线能控制 AI 幻觉风险，也能让投标文件具备审计链路：每个章节为什么这么写、响应了哪条招标要求、用了哪些公司资料、对应哪个评分点，都能回溯。

---

## 2. 目标流程

### 2.1 项目立项

用户新建投标项目时录入：

- 投标项目名称
- 投标类型，必须能映射到模板包类型
- 发包人
- 招标编号
- 截止时间
- 投标文件分册策略，例如商务标、技术标、报价标、资格审查册
- 是否允许 AI 自动补写非关键章节

系统需要立即生成项目工作流状态：

- `created`: 已创建，未上传招标文件
- `source_uploaded`: 已上传招标文件
- `analysis_running`: 招标文件解析中
- `constraints_pending_confirmation`: 约束待确认
- `outline_pending_confirmation`: 大纲待确认
- `drafting`: 初稿生成中
- `draft_reviewing`: 初稿审查中
- `revision_pending_confirmation`: 修订意见待确认
- `final_layout`: 定稿排版中
- `delivered`: 已交付
- `archived`: 已归档复盘

### 2.2 招标文件入库与拆解

输入支持 ZIP 和 PDF。ZIP 需要保留原始目录、文件名、文件类型、解析状态；PDF 需要保留页码、段落、表格、图片引用。

解析输出不应只是一段摘要，而应结构化为：

- 基本项目信息
- 资格条件
- 商务标要求
- 技术标要求
- 评分办法
- 废标条款
- 格式与排版要求
- 递交与密封要求
- 合同/规范/技术标准引用
- 附件清单
- 潜在冲突项和需要人工确认项

### 2.3 约束确认

AI 拆解结果、规范文件拆解结果、模板约束、用户人工补充共同组成“编写约束集”。约束必须有状态：

- `draft`: AI 初步抽取
- `accepted`: 用户确认有效
- `rejected`: 用户确认无效
- `merged`: 被合并到另一条约束
- `needs_review`: 低置信度或存在冲突

进入大纲融合前，至少应确认以下约束：

- 投标文件组成
- 商务标/技术标章节要求
- 评分办法
- 废标条款
- 关键格式要求
- 项目组织、人员、业绩、资质要求

### 2.4 模板包自动选择

选择逻辑建议采用明确优先级：

1. 项目 `project_type` 精确匹配模板包 `package_type`
2. 在匹配类型中选择启用状态且最新版本的模板包
3. 如果同类型存在多个模板包，按适用行业、发包人类型、历史中标率、用户偏好排序
4. 如果无法匹配，要求用户手动选择，系统不得静默使用默认模板

选择结果要写入项目，后续大纲、排版和导出均引用同一个模板包版本，避免生成过程中模板漂移。

### 2.5 模板大纲与招标解析结果融合

大纲不能简单使用模板，也不能完全让 AI 重写。建议使用“模板为骨架、招标要求为约束、评分办法为排序权重”的融合策略：

- 模板章节保留为默认骨架
- 招标文件明确要求的章节必须加入
- 评分项需要映射到至少一个响应章节
- 废标条款相关内容必须标为高优先级
- 与招标文件不一致的模板章节进入差异列表，由用户确认保留、删除、改名或移动

融合输出包括：

- 候选大纲树
- 与模板相比的新增/删除/重排/改名差异
- 每个章节对应的招标要求、评分项、规范条款
- 每个章节的生成策略：确定性插入、AI 编写、人工填写、仅排版引用
- 风险提示：缺资料、缺证明、要求冲突、不可自动生成

### 2.6 大纲确认闸门

用户确认大纲前，不启动正文初稿。原因是正文生成的上下文、评分映射、目录页码、证明材料插入都依赖稳定大纲。

确认大纲后，系统冻结一个 `confirmed_outline_version`。后续如果用户调整大纲，需要产生新版本并触发受影响章节重生成或重排。

## 3. 商务标编写建议

商务标应采用“数据装配优先，AI 辅助表达”的策略。

### 3.1 数据来源

商务标主要数据来自：

- 公司资质
- 公司资产
- 企业业绩
- 业绩证明
- 人员资质
- 人员业绩
- 法定代表人、授权委托、承诺函等固定材料
- 招标文件要求的资格审查资料

这些资料应作为结构化公司库和证据库维护，生成时根据招标约束筛选，而不是每次由用户临时上传。

### 3.2 生成方式

每个商务章节设置生成策略：

- `data_insert`: 直接插入资质、证书、业绩表、证明扫描件
- `template_render`: 根据模板变量渲染固定格式
- `ai_summary`: AI 生成公司简介、业绩亮点、响应说明
- `manual_required`: 缺少确定性数据，必须人工补充

AI 不应擅自编造证书、金额、人员履历、业绩。所有商务事实必须来自公司库或用户上传的证明材料。

### 3.3 目录与评分响应索引

商务标应生成两层索引：

1. **普通目录:** 按 Word 标题样式生成，最终排版后更新页码。
2. **评分响应索引:** 放在封面之后，列出招标评分项、响应章节、证明材料、页码位置。

页码建议在排版最终阶段处理：初稿阶段保留章节锚点，DOCX 合成后通过 Word 字段、LibreOffice 更新或自研分页结果回填页码。不要在正文生成阶段硬编码页码。

## 4. 技术标编写建议

技术标应采用“章节任务化 + 评分点驱动 + 证据约束”的策略。

### 4.1 技术标基本结构

常见技术标建议包含：

- 技术响应总表
- 项目理解与需求分析
- 总体实施方案
- 项目组织架构与岗位职责
- 进度计划与里程碑
- 资源投入计划
- 质量保证措施
- 安全、环保、文明施工或服务保障措施
- 风险识别与应急预案
- 标准规范响应
- 重点难点分析及解决措施
- 售后、运维或培训方案
- 评分点逐项响应

具体章节仍以招标文件和确认大纲为准。

### 4.2 技术标生成上下文

每个章节生成时必须传入：

- 已确认的大纲节点
- 对应招标要求
- 对应评分项
- 对应规范条款
- 项目基本信息
- 发包人和项目场景
- 可引用的公司能力、人员、设备、类似业绩
- 禁止编造的事实边界
- 模板风格和格式要求
- 上下文章节摘要，避免前后矛盾

技术标不能一次性长文生成。建议每个章节按以下过程执行：

1. 生成章节写作计划
2. 用户可选确认或调整写作计划
3. 生成章节正文
4. 对照约束进行章节自检
5. 修订并保存版本
6. 进入整册一致性审查

### 4.3 技术能力库

中长期建议建立技术能力库：

- 标准施工/服务方法
- 行业解决方案
- 质量、安全、进度、风险管理模板
- 项目组织岗位职责模板
- 常用设备和资源配置模板
- 类似项目复用段落
- 中标后沉淀的优秀章节

初期可以先从已有历史投标文件和用户确认稿中抽取，形成可检索片段库。AI 编写技术标时优先引用能力库内容，再结合招标文件定制化改写。

## 5. 组织架构等图表生成建议

图表不要让 AI 直接“画图”。推荐做成结构化图表规格，再由程序稳定渲染。

### 5.1 图表类型

第一阶段建议支持：

- `org_chart`: 项目组织架构图
- `process_flow`: 实施流程图
- `schedule_gantt`: 进度计划甘特图
- `responsibility_matrix`: 岗位职责矩阵
- `risk_matrix`: 风险分级与应对矩阵

### 5.2 图表生成链路

图表链路建议为：

1. 从招标约束提取岗位、工期、流程、质量安全要求
2. 从公司库选择人员、角色、资质和类似业绩
3. 生成 `chart_spec_json`
4. 用户预览并调整节点、层级、文字
5. 程序渲染 SVG 或 PNG
6. 嵌入 DOCX，并保存为可复用图表资产

示例规格：

```json
{
  "type": "org_chart",
  "title": "项目组织架构图",
  "nodes": [
    {"id": "pm", "label": "项目经理", "person": "张三", "qualification": "一级建造师"},
    {"id": "tech", "label": "技术负责人", "person": "李四"},
    {"id": "quality", "label": "质量负责人", "person": "王五"}
  ],
  "edges": [
    {"from": "pm", "to": "tech"},
    {"from": "pm", "to": "quality"}
  ],
  "style": {
    "orientation": "vertical",
    "theme": "bid_formal"
  }
}
```

这样做的好处是：图表可审查、可编辑、可重渲染、可嵌入 Word，不依赖不可控图片生成。

## 6. 审查、评分与修订闭环

初稿完成后建议分两轮审查：

### 6.1 初步评审

初步评审目标是发现硬伤：

- 是否遗漏招标文件要求
- 是否触发废标条款风险
- 商务资质是否缺失
- 技术方案是否明显不符合项目场景
- 是否与规范文件冲突
- 是否存在常识性错误
- 是否存在 AI 编造事实

输出为问题清单，按 P0/P1/P2/P3 分级。

### 6.2 评分对照复审

用户确认初步修改方向后，进行评分对照复审：

- 每个评分项对应哪些章节
- 当前文本预计可得分区间
- 缺少哪些证明或技术亮点
- 哪些章节需要加强响应
- 修订建议是否已落实

评分建议不应承诺真实得分，只能作为内部质量评估。

### 6.3 修订追踪

每条审查意见要能追踪：

- 来源约束或评分项
- 问题章节
- 严重级别
- 建议修改
- 用户处理决定
- 修订状态
- 修订前后版本

只有 P0/P1 关闭或被用户明确接受风险后，才允许进入定稿排版。

## 7. 排版、导出与归档

### 7.1 排版

最终排版应使用模板包和招标格式要求共同控制：

- 封面
- 目录
- 评分响应索引
- 标题层级
- 页眉页脚
- 页码
- 表格样式
- 图片尺寸
- 分册规则
- 签章/盖章占位

排版阶段生成 DOCX，再由 DOCX 转 PDF。PDF 应作为最终投递预览，DOCX 保留可编辑版本。

### 7.2 分割与打包

支持：

- 整册 DOCX/PDF
- 按商务标、技术标、报价标分册
- 按章节拆分
- ZIP 打包下载
- 附件和证明材料单独打包

### 7.3 最终版归档

定稿后需要存储：

- 最终 DOCX
- 最终 PDF
- 分册文件
- 生成时使用的模板包版本
- 确认大纲版本
- 约束集版本
- 审查问题闭环记录
- 用户最终确认时间

开标后补充：

- 是否中标
- 投标报价与排名
- 评分详情
- 失败原因或中标关键因素
- 后续复用建议

这些数据用于优化后续模板选择、技术能力库、评分响应策略。

## 8. 当前代码基础与改造点

现有系统已经具备以下基础：

- 项目表已有投标类型等基础字段
- 招标文件上传和解析链路
- AI 需求抽取和人工审核状态
- 模板包与模板项导入、渲染能力
- 投标大纲生成基础服务
- 章节生成基础服务
- 审查问题与交付包基础表
- DOCX 导出基础能力

主要缺口：

- 发包人、模板选择、工作流状态未形成完整项目设置闭环
- 投标类型到模板包类型的自动选择缺少统一服务
- 招标解析约束、规范约束、模板约束没有统一版本化
- 模板大纲与招标要求的差异融合和用户确认闸门不完整
- 商务标和技术标缺少不同生成策略
- 组织架构图、流程图、甘特图等图表资产缺少结构化生成
- 审查和评分复审还需要更强的约束映射和修订追踪
- 最终文件归档和开标后复盘数据模型不完整

## 9. 建议新增/强化的核心服务

- `ProjectSetupService`: 项目元数据、投标类型、发包人、工作流状态管理。
- `TenderConstraintService`: 招标解析结果、规范条款、用户确认约束的统一管理。
- `TemplateSelectionService`: 根据投标类型自动选择模板包。
- `OutlineReconciliationService`: 模板大纲与招标要求融合，输出差异和确认版本。
- `BusinessBidAssembler`: 商务标资料筛选、证据插入、评分索引生成。
- `TechnicalBidWriter`: 技术标逐章计划、生成、自检、修订。
- `ChartGenerationService`: 图表规格生成、预览、渲染、嵌入。
- `BidReviewService`: 初审、评分复审、修订意见闭环。
- `FinalDeliveryService`: DOCX/PDF 排版、分册、打包、归档。
- `PostBidReviewService`: 开标结果、盘点、历史学习。

## 10. 分阶段落地建议

### 第一阶段：把流程跑顺

目标是确保从新建项目到确认大纲、生成商务/技术初稿、导出交付包可以闭环。

范围：

- 项目元数据补齐
- 模板自动选择
- 约束确认
- 大纲差异融合和确认
- 商务标资料插入基础版
- 技术标逐章生成基础版
- 初步审查
- DOCX/PDF 导出

### 第二阶段：提高质量和可控性

目标是减少 AI 幻觉，提高评分响应质量。

范围：

- 评分响应索引
- 章节级引用追踪
- 图表结构化生成
- 技术能力库
- 修订追踪
- 二轮评分复审

### 第三阶段：形成投标知识飞轮

目标是用历史结果指导未来投标。

范围：

- 中标/未中标复盘
- 模板命中率分析
- 评分项改进建议
- 历史优秀章节复用
- 发包人偏好沉淀

## 11. 风险与控制

- **AI 编造事实:** 商务标事实必须来自公司库或证明文件，技术标事实引用必须可追踪。
- **大纲频繁变更:** 使用大纲版本和影响章节标记，避免全量重生成。
- **页码不准确:** 页码只在最终排版阶段回填，初稿使用锚点。
- **图表不可编辑:** 使用结构化规格和程序渲染，避免纯图片黑盒。
- **模板与招标要求冲突:** 差异必须展示给用户确认，系统不得自动吞掉冲突。
- **最终文件不可复盘:** 保存最终包、约束版本、大纲版本、审查记录和开标结果。
