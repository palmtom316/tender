# 导入方法

## 总体流程

推荐将导入分为两步：

1. 导入结构化主数据。
2. 导入附件并挂靠到主数据。

这样做可以避免附件先入库但找不到 `owner_id`。

## 导入前准备

1. 确认 `companybase/templates/companybase_master.xlsx` 已复制到 `companybase/imports/` 并填写。
2. 确认所有附件文件都在 `companybase/files/` 下。
3. 运行离线校验：

```bash
python3 companybase/tools/validate_companybase.py companybase/imports/companybase_master_YYYY-MM-DD.xlsx
```

4. 修复校验报告里的 P0/P1 问题。

## 幂等导入顺序

导入器应按以下顺序执行：

1. `公司主体` → `library_company`
2. `公司资料` → `company_profile`
3. `企业资质` → `qualification_certificate`
4. `人员资料` → `person_profile`
5. `公司资产` → `company_asset`
6. `项目业绩` → `project_performance`
7. `财务报表` → `financial_statement`
8. `制度能力` → 可导入 `evidence_asset` 或后续能力表
9. `附件索引` → `evidence_asset`

## API 导入映射

现有系统已支持以下接口，可由导入脚本调用：

```text
GET/POST /api/master-data/library-companies
GET/POST /api/master-data/people
GET/POST /api/master-data/library-companies/{library_company_id}/assets
POST     /api/master-data/evidence-assets/upload
PUT      /api/master-data/evidence-assets/{id}
POST     /api/master-data/evidence-assets/{id}/replace-file
```

资质、业绩、财务、公司资料对应接口在系统中已有 repository；若 API 未暴露完整批量导入接口，建议新增一个后台导入命令或管理端 API，不建议手工逐条录入。

## 推荐导入器行为

### 主数据

使用 `unique_key` 写入 `metadata_json.import.unique_key` 或 `profile_json.import.unique_key`。导入时先按 unique_key 查询已有记录：

- 不存在：新增。
- 存在：更新可变字段。
- 标记 `delete_flag=TRUE`：不物理删除，建议更新状态为停用/退役。

### 附件

导入器对 `file_relative_path` 做：

1. 路径必须在 `companybase/` 内。
2. 文件必须存在。
3. 文件类型必须属于 PDF/图片白名单。
4. 计算 SHA256。
5. 查找同一 `owner_type + owner_unique_key + asset_type + sha256` 是否已存在。
6. 已存在则跳过；不存在则上传或复制后创建 `evidence_asset`。

附件导入建议写入 `metadata_json`：

```json
{
  "import": {
    "source": "companybase",
    "attachment_key": "att_cert_safety_license_pdf",
    "owner_unique_key": "cert_safety_production_license",
    "file_sha256": "...",
    "import_batch": "2026-05-13"
  },
  "blind_bid": {
    "sensitive": true,
    "redaction_note": "含公司名称和证书编号，暗标正文不得直接输出"
  }
}
```

## 失败处理

导入器不应遇到第一条错误就中断整批。建议输出三类报告：

- `import_success.csv`：新增/更新/跳过记录。
- `import_errors.csv`：P0 错误，如 owner 找不到、文件不存在、JSON 无效。
- `import_warnings.csv`：P1/P2 风险，如证书即将到期、附件暗标敏感、缺核心附件。

## 开发库与上线库复用

不要把开发库导出 SQL 作为上线资料源。推荐上线流程：

1. 开发库用同一套 `companybase` 资料包导入，验证字段和附件。
2. 修正 Excel 和附件目录。
3. 给 `companybase` 打版本备份。
4. 上线库使用同一资料包重新导入。
5. 比对导入报告记录数、附件 hash 和关键字段。

## 后续可开发的导入命令

建议命令形态：

```bash
python3 backend/tender_backend/tools/import_companybase.py \
  --xlsx companybase/imports/companybase_master_2026-05-13.xlsx \
  --files-root companybase \
  --mode upsert \
  --dry-run

python3 backend/tender_backend/tools/import_companybase.py \
  --xlsx companybase/imports/companybase_master_2026-05-13.xlsx \
  --files-root companybase \
  --mode upsert
```

`--dry-run` 必须只校验不写库。
